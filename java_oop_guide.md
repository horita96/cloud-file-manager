# 🏗️ 1. クラスとインスタンス

オブジェクト指向のすべての基本。プログラムを「型」として定義することと、それを「メモリ上で動かす」ことの分離。

---

## 🌎 そもそも「オブジェクト指向」とは？
一言でいうと、**「現実世界の仕組み（モノや役割分担）を、そのままプログラムの中に持ち込んで組み立てる開発スタイル」**。
データと処理を『モノ（オブジェクト）』として小分けにグループ化し、部品を組み立てるようにアプリを作ることで、コードの整理整頓（保守性の向上）を目指す知恵です。

---

## 📌 【基本】文法と書き方
* **クラス（設計図）**: `class` キーワードを使って、オブジェクトが持つべき「データ（フィールド）」と「振る舞い（メソッド）」を定義する。
* **インスタンス（実体）**: `new` キーワードを使って、設計図から実体（モノ）をメモリ上に生み出す。

```java
// 1. 設計図を定義する（クラス）
public class Hero {
    String name; // フィールド（属性/データ）
    int hp;      // フィールド

    void attack() { // メソッド（操作/振る舞い）
        System.out.println(this.name + "の攻撃！");
    }
}

// 2. 別の場所で実体化して使う（メイン処理）
public class Main {
    public static void main(String[] args) {
        Hero h1 = new Hero(); // ★インスタンス化（実体化）
        h1.name = "アベル";
        h1.hp = 100;
        h1.attack(); // ➔ 「アベルの攻撃！」と動く
    }
}
```

🧠 【詳細】メモリ（JVM）の内部挙層
メタスペース（静的領域）: プログラム起動時にクラスの「構造情報」や「メソッドの手順（筋肉）」が配置される不変の空間。

ヒープ領域（動的領域）: new された瞬間に、実データの箱がガツンと確保される空間。

⚠️ 【超重要】参照型の罠
Java
Hero h1 = new Hero("アベル"); // メモリの100番地にインスタンスが作られ、h1には「100」というアドレスが入る
Hero h2 = new Hero("アベル"); // メモリの200番地にインスタンスが作られ、h2には「200」というアドレスが入る
名前やHPが全く同じインスタンスであっても、new を2回行えばメモリの場所（アドレス）が異なるため、h1 == h2 の比較結果は false になる。

# ⚡ 2. コンストラクタ

インスタンスが**生まれた最初の1瞬だけ自動で動く特別なメソッド**。不完全な状態のオブジェクトがシステム内に誕生するのを文法レベルで防ぐ。

## 📌 【基本】文法と書き方
* メソッド名が**「クラス名と完全に同じ」**であり、**「戻り値（voidなど）を書かない」**のが絶対ルール。
* `new` したときに自動で呼び出され、初期データを安全にセットできる。


```java
public class Hero {
    String name;
    int hp;

    // ★これがコンストラクタ（初期化の儀式）
    public Hero(String name) {
        this.name = name; // 生まれた瞬間に名前を強制的にセット
        this.hp = 100;    // 初期HPを100に自動固定
    }
}
```

🧠 【詳細】コンストラクタチェイン（バケツリレー）
実務では、利用シーンに応じて複数のコンストラクタを用意（オーバーロード）するが、それぞれの処理にバラバラに初期化を書くと修正漏れバグが起きるため、this() を使って一箇所に集約する。

```java
public class Hero {
    String name;
    int hp;

    public Hero() {
        this("名無しの勇者"); // 名前なしで生まれたら、この文字で下のコンストラクタへリレー
    }
    
    // ★ここに初期化ロジックを一箇所に集約（マスターコンストラクタ）
    public Hero(String name) {
        this.name = name;
        this.hp = 100;
    }
}
```

⚠️ 絶対ルールと仕様の罠
1行目の鉄則: this() や super() によるリレー命令は、コンストラクタの絶対に1行目に書かなければならない。

デフォルトコンストラクタの消滅: クラス内に引数つきコンストラクタを1つでも手動で書いた瞬間、Javaが自動付与してくれていた「空っぽのコンストラクタ（引数なし）」は消滅する。

# 🌿 3. 継承（extends）

既存のクラスの遺伝子を引き継ぎ、**「変更したい部分（差分）だけを書いて新しいクラスを作る」**仕組み。

## 📌 【基本】文法と書き方
* `extends` キーワードを使い、`子クラス extends 親クラス` の形で記述する。

```java
// 親クラス（ベース）
public class Matango {
    char suffix;
    public void attack(Hero h) {
        System.out.println("マタンゴ" + this.suffix + "の攻撃！");
    }
}

// 子クラス（派生）
public class PoisonMatango extends Matango {
    // ★Matangoのsuffixやattack()は書かなくても自動で引き継がれている！
    int poisonCount = 5; // ポイズン特有のデータを1行だけ追加（差分開発）
}
```

🧠 【詳細】親子ドッキングとオーバーライド
メモリ上の仕組み: 子クラスを new した時、ヒープ領域内ではまず内側に「親のインスタンス」が生成され、その外側を「子のインスタンス」の皮が包み込むようにドッキングして1つの巨大なオブジェクトになる。そのため、先に親を初期化する super() が内部で絶対ルールとして呼び出される。

Is-a関係の厳格化: 「子は親の一種である（SuperHero is a Hero）」という厳格な関係が成り立つ場合のみ継承を使う。

🛠️ 特権：オーバーライド（技の上書き）
親から引き継いだメソッドの動きを、子クラス側で自分の都合に合わせて書き換えること。

メソッドの「名前」と「引数」は完全に一致させる必要がある。

頭に @Override（アノテーション） をつけることで、タイポ（打ち間違い）によるバグをコンパイラに検知させるのが実務の鉄則。

# 🔒 4. カプセル化（Encapsulation）

**「クラスのメンバーに南京錠をかけ、外部からの不正な書き換えをシャットアウトするセキュリティ機構」**（ホームセキュリティ）。

## 📌 【基本】文法と書き方
* **隠す**: フィールドの頭に `private` をつけ、外から直接触らせない。
* **窓口（ゲッター/セッター）**: 公開された（`public`）メソッド経由でしか読み書きできないようにする。

```java
public class Hero {
    private int hp; // 🔒 privateにより「h.hp = -500;」と直接触るとコンパイルエラーになる

    // 🚪 変更用窓口（セッター）
    public void setHp(int hp) {
        if (hp < 0) {
            this.hp = 0; // 🛑 安全フィルター：マイナス値が来たら強制的に0にする
            return;
        }
        this.hp = hp;
    }
    
    // 🚪 読み出し用窓口（ゲッター）
    public int getHp() {
        return this.hp;
    }
}
```

🧠 【詳細】なぜ実務で命より大事とされるのか？
カプセル化をしていないと、誰かが埋め込んだ「HPがマイナスになるバグ」を探すために、プロジェクト内の何万行ものコードを全調査することになります。
カプセル化をしておけば、データが変わる瞬間は絶対にこの setHp メソッドを通るため、「バグの原因は、Mainではなく、このクラスのセッターの中にしかない」と一瞬で特定でき、アプリの保守性・安全性が爆上がりします。

# 🎭 5. 多態性（ポリモーフィズム）

**「指示を出す側（Mainなど）が、相手の具体的な型を気にせず、共通の親の型として『雑に、まとめて命令できる』魔法」**。

## 📌 【基本】文法と書き方
* Javaの強力なルール「**子クラスのインスタンスは、親クラスの型の箱（変数や配列）に代入して良い**」を活用する。

```java
// ★異なる職業（子）を、すべて親である「Character型」の配列にまとめて格納する
Character[] party = { new Hero(), new Wizard(), new Thief(), new Ninja() };

for (Character ch : party) {
    ch.attack(); // 📢 全員に対して、全く同じ「attack」という命令を投げるだけ！
}
```

🧠 【詳細】動的結合（ダイナミックバインディング）
Main側は相手の正体を一切気にせず ch.attack() と叫んでいるだけ。しかし、Javaの実行エンジンが実行されたその一瞬でメモリ上の実体を確認し、

魔法使いなら ➔ Wizardがオーバーライドした魔法攻撃

忍者なら ➔ Ninjaがオーバーライドした手裏剣攻撃へと、自動で適切なメソッドへ案内してくれる。
